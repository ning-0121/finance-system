## A. Provenance（来源链）

### A.1 现状

| 实体 | source_type/id 字段 | 审计表覆盖 | 评分 |
|---|---|---|---|
| `journal_entries` | ✓ source_type + source_id + source_document_id | 不在 audit_log；用 `status='voided'` 标记+`reverse_gl_on_void`触发器 | 良 |
| `budget_orders` | ✗ 无 source 字段 | `financial_audit_log` 覆盖 4 个字段（target_purchase_price, estimated_freight, estimated_commission, estimated_customs_fee）| **不足** |
| `payable_records` | ✓ budget_order_id + settlement_id + invoice_id | **零字段级审计** | **缺失** |
| `actual_invoices` | ✓ budget_order_id + sub_document_id | **零字段级审计** | **缺失** |
| `order_settlements` | ✓ budget_order_id | **零字段级审计** | **缺失** |
| `financial_risk_events` | ✓ related_order_id/customer_id/supplier_name | **零字段级审计** | **缺失** |
| `entity_freezes` | ✓ 有 freeze_reason / trigger_source | 有自身字段（frozen_by/unfrozen_by）| 中 |
| `cost_items` | ✓ source_module + source_id | 软删除标记 ✓；无字段级审计 | 中 |

实测 `financial_audit_log`：仅有 5 条历史记录，全部针对 `budget_orders` 的 4 个 numeric 字段。**支付/回款/凭证/决算的字段变更从未被审计过**。

### A.2 缺陷

- **CFO 7 问无法回答**：拿任意一张 `payable_records.payment_status='paid'` 记录，目前无法回答：
  - "谁批准的"？只能从 `approved_by` 字段读 user_id，但**没有时间线**——multiple state transitions 全压在 `approved_at/paid_at` 两个时间戳，中间细节丢失
  - "依据是哪张发票/合同"？只有 `invoice_id` 一个 FK，无 OCR/document 链
  - "为什么这个金额"？没有计算来源（人工录入还是从 invoice OCR 来的）

- **`trust_scores` 表不存在**：之前的代码注释和 todos 反复提到 trust score 降级逻辑，但 schema 没建该表。所以"AI 自动写入需 trust 达标"的策略**目前是空话**。

- **`journal_provenance` / `financial_provenance` 表都不存在**：你这次任务里要求建立的来源链结构，**DB 一片空白**。

### A.3 修复建议（待你审批）

**Migration `20260516_provenance_layer.sql`**（草案 — 后文给出完整 SQL）：
1. 创建 `financial_provenance` 表：每条记录答 7 问（actor / reason / source_kind / source_id / approved_by / reversal_target_id / affected_reports[]）
2. 给 5 张缺失审计的财务表加 AFTER INSERT/UPDATE/DELETE → financial_provenance 触发器
3. 创建 `trust_scores` 表（actor × entity_type → score / confidence / last_downgrade_at / reason）
4. 在 `create_journal_atomic` RPC 内自动写一行 provenance（避免应用层忘记）

**Rollback 风险**：低（纯 ADD，触发器可独立 DROP）。
**Test plan**：插入测试订单 → audit 表应有 N 条记录覆盖所有字段；voided 凭证应被 reverse_gl_on_void 同步反向 provenance。

## B. Freeze Propagation（冻结传播）

### B.1 现状

**结构**：`src/lib/engines/freeze-engine.ts` + `entity_freezes` 表 + `src/lib/engines/submit-gate-engine.ts`（前端提交时检查）

**实测**：
```
1) INSERT entity_freezes(entity_type='budget_order', entity_id=X, status='frozen')
2) RPC create_journal_atomic(source_id=X, ...)
   → 结果: 凭证 PZ-202605-0001 创建成功 ❌
```

### B.2 缺陷（P0 级）

**`entity_freezes` 是装饰品**。它只在以下 2 个路径被检查：
- `src/lib/engines/submit-gate-engine.ts:261` —— UI 提交时
- `src/lib/engines/orchestration-engine.ts:202` —— Agent 编排引擎冲突检测

**没有任何路径强制 freeze**：
- ✗ `create_journal_atomic` RPC 不检查
- ✗ `payable_records` 状态机触发器不检查
- ✗ `cost_items` 插入不检查
- ✗ API 路由（如 `/api/orders/[id]/settlement`）不检查
- ✗ Webhook (`/api/integration/webhook`) 自动建单不检查
- ✗ Cron / 期末关账（`closing-engine.ts`）不检查

**业务后果**：财务经理冻结一笔可疑订单后，**只要绕过 UI（API 直调 / agent 自动化 / 期末批量过账）一样会把它写进 GL**。这是 SOX 类合规的根本性漏洞。

### B.3 修复建议

**最小化加固方案**（推荐）：
1. 给 5 张关键表（journal_entries / payable_records / cost_items / order_settlements / actual_invoices）加 `BEFORE INSERT/UPDATE` 触发器，调用 `check_entity_not_frozen(entity_type, entity_id) RETURNS void`，如果冻结则 `RAISE EXCEPTION`
2. `create_journal_atomic` RPC 内加同一份检查（lines 里每个 order_id 都查一遍）
3. 解冻流程需走 approval：`entity_freezes.unfreeze_requested_at` → 二级审批 → `unfrozen_by` 必须 ≠ `frozen_by`

**Rollback 风险**：中。如果触发器误判（如订单本不该冻结但被错冻），所有后续操作被阻塞。需要：
- `BYPASS_FREEZE` 临时角色（emergency override + 强制写 audit）
- 解冻必须可逆且留痕

**Test plan**：复杂场景测试中加场景 9 "冻结传播"——冻结订单后尝试 6 种写入（cost / AP / 付款 / 收款 / GL / 决算），全部应被拒绝并附冻结原因。

## D. Mutation Safety（变更安全审计）

### P0 — 数据损坏 / 静默错误

**P0-1 · 非原子 GL 写入（孤儿凭证风险）**
- **文件**：`src/lib/accounting/fx-gains.ts:105-135`
- **Root cause**：先 `INSERT journal_entries` 拿到 id，再 `INSERT journal_lines` 用该 id。两步分离，**未走 `create_journal_atomic` RPC**。
- **Impact**：网络抖动 / DB 重启时 header 已写而 lines 失败，产生借贷为 0 / 行为空的孤儿凭证。期末汇兑结转尤其常见。
- **Reproduction**：mock supabase client 在 lines insert 时抛错 → 留下空 header。
- **Fix strategy**：改写为 `supabase.rpc('create_journal_atomic', { p_lines: ... })`。10 行代码改动。
- **Rollback risk**：极低。
- **Test plan**：单元测试 mock 失败 + E2E 跑期末关账。

**P0-2 · 6 张财务表缺 `deleted_at` 列 → softDelete 静默降级为硬删除**
- **文件**：`src/lib/save-guard.ts:130-160` + DB schema
- **Root cause**：`SOFT_DELETE_TABLES = {actual_invoices, cost_items, payable_records}` 列表中，**只有 cost_items 真的有 `deleted_at` 列**。`actual_invoices` 和 `payable_records` 缺列，softDelete 调用 UPDATE 失败时 `console.warn` 后**调 `safeDelete` 物理删除**。`order_settlements / budget_orders / shipping_documents / financial_risk_events` 根本不在 SOFT_DELETE_TABLES 里，永远硬删。
- **Impact**：财务记录被永久物理删除，无审计、无追溯、无法重建。
- **Reproduction**：UI 点删除发票 → DB 直接物理删除（实测 `deleted_at` 列不存在）。
- **Fix strategy**：迁移加 6 列 `deleted_at timestamptz / deleted_by uuid / deletion_reason text`；扩展 SOFT_DELETE_TABLES；所有读查询加 `.is('deleted_at', null)` 过滤。
- **Rollback risk**：中（读查询要全量加过滤，遗漏会显示已删除数据）。
- **Test plan**：每张表的 read query 都要单测验证软删除记录不可见；删除后查 audit 链应能 trace。

**P0-3 · 冻结传播形同虚设**
- 见 §B.2 详述。Repro 已做（实测冻结订单仍可过账）。
- **Fix**：见 §B.3 触发器方案。

**P0-4 · 直接硬删 cost_items**
- **文件**：`src/app/(main)/costs/page.tsx:637`
- **Root cause**：`supabase.from('cost_items').delete().eq('id', item.id)` 绕过 `safeguard.softDelete`。即使 cost_items 表已经有 `deleted_at` 列，此处也物理删除。
- **Impact**：人工误删 → 已结算订单的成本归集丢失 → 毛利率不准。
- **Fix**：改为 `import { softDelete } from '@/lib/save-guard'` 并强制要求 reason。
- **Rollback risk**：低。
- **Test plan**：UI 单测验证软删除后该行从列表消失但 DB 可查到 deleted_at。

**P0-5 · rollback API 白名单引用不存在的表**
- **文件**：`src/app/api/documents/rollback/route.ts:49-54`
- **Root cause**：`ROLLBACK_ALLOWED_TABLES` 含 `'receivable_records'` 和 `'payment_records'`，但**这两张表在 schema 中根本不存在**（cross-check 已确认）。回滚请求会调用 `supabase.from('receivable_records').delete()` 静默失败（0 rows affected），返回 success。
- **Impact**：使用错误表名的回滚动作显示"已回滚"，但实际什么都没做。回滚函数 → 用户以为还原了 → 财务数据不一致。
- **Fix**：移除 ghost tables；改为白名单只包含**经过验证存在的表**；删除前查 `pg_tables` 或维护一份硬编码 + 单测校验的清单。
- **Rollback risk**：低。
- **Test plan**：单元测试枚举白名单中每张表跑 `SELECT 1 FROM <table> LIMIT 0`，存在即通过。

### P1 — 高风险

**P1-1 · `order_settlements.status` 无 DB 触发器**
- **文件**：`src/lib/supabase/schema-v2.sql:113`
- **Impact**：理论上可 `UPDATE order_settlements SET status='draft' WHERE status='locked'` 把已锁单恢复编辑。当前唯一约束是 UI 检查（`api/orders/[id]/settlement/route.ts:33`）。
- **Fix**：加 `trg_settlement_status` 触发器，参照 `trg_payable_status_fn` 模式。状态机：`draft → confirmed → locked`，locked 终态。

**P1-2 · `journal_entries.status` 缺前置 transition 校验**
- **文件**：`migrations/schema-p1-fixes.sql:68-97`
- **Root cause**：现有 `enforce_journal_immutability()` 只挡 lines 修改，不挡 header.status 任意跳转。理论上 `UPDATE journal_entries SET status='draft' WHERE status='posted'` 会让 reverse_gl_on_void 不触发（只有 posted→voided 才触发），余额异常。
- **Fix**：加 BEFORE UPDATE OF status 校验：合法转换 {draft→posted, posted→voided}。

**P1-3 · `webhook` 自动建单 silent catch**
- **文件**：`src/app/api/integration/webhook/route.ts:156`
- **Code**：`} catch (err) { console.error('自动创建预算单失败:', err) }`
- **Impact**：外部 ERP 推订单进来 → synced_orders 写成功 → budget_orders 静默失败 → 销售看到"订单已同步"但财务系统里没单。
- **Fix**：失败路径写入 `save_diagnostic_logs` (status='error') + 向 synced_orders 标记 sync_status='draft_failed'。

**P1-4 · `payable_records / actual_invoices / order_settlements` 状态变更零字段级审计**
- 见 §A.2
- **Fix**：见 A.3 的 provenance migration

**P1-5 · `queries-v2.ts:241` upsert 决算无乐观锁**
- 并发决算重算会 silently 覆盖。加 version 列。

**P1-6 · 多个 demo 数据 fallback（你已部分修过）**
- queries.ts 上轮提交已修。但 `src/lib/supabase/queries-v2.ts` 是否同样？需复查。
- **Action**：grep queries-v2.ts 重复执行类似审计。

**P1-7 · closing-engine 自动过账无 trust/confidence 门控**
- **文件**：`src/lib/engines/closing-engine.ts:516`
- **Root cause**：调用 `create_journal_atomic` 写期末凭证，没有任何"AI confidence ≥ X / no anomaly / approval"前置检查。
- **Impact**：违反"AI Never Posts Final Accounting Entries Autonomously"原则。
- **Fix**：所有自动调用 RPC 之前，先查 `entity_freezes` + `financial_risk_events`（active pending）+ `trust_scores`（待建表）。若任一不满足，存为 `journal_entries.status='draft'` 而非 'posted'，等人审。
