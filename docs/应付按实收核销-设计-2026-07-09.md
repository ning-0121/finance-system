> 关联：全链路审计 2026-07-09 P1 #3；已上线 **A 止血**（采购单工作台登记前显示实收/订购、短装提示按实收付款，commit fa908c6）
> 目标读者：财务系统负责人 + 会计口径决策人

## 1. 问题（审计结论）

**收货实收数据到达财务后从不回冲应付 → 应付永远按理论下单量，短装即系统性多付。**

现状数据流：

```
采购下单(节拍器) → purchase_order.placed → fin_purchase_orders / fin_po_lines(ordered_qty, unit_price, amount)
收货(节拍器)     → goods_receipt.recorded → fin_po_lines.received_qty / inspection_result  ← 【写入后无人读】
应付             → 人工在「采购单工作台」把 PO 下单额(total_amount)登记为 cost_items(fin_status='registered')
付款             → payables 页 unpaid = Σcost_items − 付款(按供应商名匹配)
```

- `fin_po_lines.received_qty`（migration 20260705）**已被 webhook 正确写入，但全仓无任何读取点**（死写）。
- 真实应付 = `cost_items`，来自人工登记的**下单额**，与实收（received_qty）**零关联**。
- 短装/超收/让步/拒收都不驱动应付调整 → 规模化后必然错付/漏冲。

**A 止血已做**：登记前把「实收/订购」显示出来、短装提示按实收付款。但仍靠人肉，且已登记的单不追溯。B 才是"按实收核销"的真解。

---

## 2. 目标与原则

**目标**：应付以**已验收合格实收**为准自动核销，短装少付、超收（经审批）多付、拒收不付，全程留痕、可回滚，且**符合现有 GL 双分录口径**（journal_entries / atomic_journal RPC）。

**原则**（沿用 AGENTS.md 铁律）：
1. **财务不自造数** —— 实收数字全来自节拍器 `goods_receipt.recorded`（源头是仓库/QC 人工录），财务只据此核销，不凭空改。
2. **合格口径** —— 只有 `inspection_result ∈ {pass, concession}` 计入实收；`reject` 不计（节拍器侧已在 P1 #4 修为回写 received_qty 排除 reject，此处双保险再校验）。
3. **人确认放款** —— 自动算出"按实收应付"，但**付款动作仍需人点**；系统只做**闸与提示**，不自动打款。
4. **留痕可回滚** —— 任何应付调整走 journal_entries（红冲/补记），有 source 溯源，可反向。
5. **不破坏已上线** —— A 止血保留；B 增量叠加，灰度可回退。

---

## 3. 核心口径

对每张采购单（fin_purchase_order），按其 fin_po_lines 派生三个金额：

| 口径 | 定义 | 用途 |
|---|---|---|
| **订购应付 ordered_payable** | Σ `amount`（= ordered_qty × unit_price） | 下单即知，登记时的 provisional 应付 |
| **实收应付 received_payable** | Σ (合格 received_qty × unit_price)，合格 = inspection_result≠reject | **核销权威**：收齐后应付应等于它 |
| **已登记应付 registered_payable** | 已入 cost_items 的金额 | 当前真实台账值 |

**核销差额 = registered_payable − received_payable**：
- `> 0` **短装**：多登记了，需**红冲差额**（少付）。
- `< 0` **超收**：少登记了，超收部分**需财务审批**后**补记**（多付）。
- `= 0` 一致，无需动作。

> 注：仅当该单**收货已完成**（所有行都有 received_at，或 PO status=received/closed）才做终态核销；收货途中只提示、不冲销（避免分批收货中途反复红冲）。

## 4. 方案对比

### B1（推荐）：登记保持下单额（provisional）+ 收货完成时终态核销
- 登记为费用：仍记 `ordered_payable`（下单额），置 cost_item.reconcile_status='provisional'。**流程零改动，兼容存量**。
- 收货完成事件（新增 or 复用 goods_receipt.recorded 判 PO 收齐）→ 计算 received_payable → 若与 registered 差额超阈值：
  - 短装：对该 cost_item 发一笔**红冲** journal（贷记应付、冲减差额），cost_item.reconcile_status='reconciled_short'，记 note「按实收核销，短装 X」。
  - 超收：**不自动补记**，置 reconcile_status='over_pending_review'，进「超收待审批」列表；财务审批放行后再补记 journal。
- 付款闸：付款金额 > received_payable（且非超收已批）→ 拦截/二次确认。

**优点**：登记流程不变、存量单可批量跑一次核销、红冲留痕、超收有闸。
**缺点**：需要一个可靠的"收货完成"信号（见 §6）。

### B2：登记时即按 min(已收, 已订) 入账
- 登记为费用时，金额直接取 `min(received_payable, ordered_payable)`。
- 缺点：登记常发生在**收货前**（received=0 → 应付=0），不符合业务（下单就该挂 provisional 应付）；且后续到货仍要补记。**不推荐单独用**，可作为 B1 的"登记时若已收齐则直接按实收"优化。

### B3（完整，留给大版本）：建 fin_goods_receipts 明细表 + 应付按行级实收重算 + 差异台账
- 收货明细独立成表，应付按 line 级实收滚动重算，支持分批、退货冲回、多币种精确到行。
- 工作量大、接近应付重构，**本设计不展开**，作为 SCM/应付大版本目标。

**结论：采纳 B1**（+ B2 的"登记时已收齐直接按实收"作为优化点）。

## 5. 数据模型改动（B1）

均为**加列/加状态**，不改现有列语义：

```sql
-- cost_items：核销状态 + 溯源到采购单 + 实收核销留痕
ALTER TABLE public.cost_items
  ADD COLUMN IF NOT EXISTS fin_po_id           uuid REFERENCES public.fin_purchase_orders(id),
  ADD COLUMN IF NOT EXISTS reconcile_status     text NOT NULL DEFAULT 'none'
    CHECK (reconcile_status IN ('none','provisional','reconciled_short','over_pending_review','reconciled_over')),
  ADD COLUMN IF NOT EXISTS ordered_payable      numeric(15,2),   -- 登记时下单额快照
  ADD COLUMN IF NOT EXISTS received_payable     numeric(15,2),   -- 最近一次核销的实收应付
  ADD COLUMN IF NOT EXISTS reconciled_at        timestamptz;

-- fin_po_lines 已有 received_qty/inspection_result/received_at(20260705),无需加列。
-- 收齐判定用派生视图或函数(见 §6),不落冗余标志。
```

> 若 cost_items 目前未直接绑 fin_po（靠供应商名匹配），`fin_po_id` 回填是本设计的关键前置：登记为费用时把当前 PO 的 id 写进去（登记入口已持有该 PO，改动小）。存量 cost_items 靠 po_no/供应商名一次性回填脚本。

## 6. "收货完成"信号（B1 依赖）

三选一（按可靠性排序）：
1. **节拍器发终态事件**（最干净）：节拍器在一张采购单所有执行行收齐/关单时，发 `purchase_order.received`（或复用 goods_receipt.recorded 带 `po_closed:true`）。财务据此触发终态核销。**需节拍器侧配合发送**。
2. **财务侧按 fin_po_lines 自判**：收到每条 goods_receipt.recorded 后，检查该 fin_po 的所有 line 是否都有 received_at 且 Σreceived≥Σordered×(1−短装容差) → 视为收齐 → 核销。**纯财务侧、无需节拍器改**，但"收齐"判定可能滞后于真实关单。
3. **人工触发**：应付页给短装单一个「按实收核销」按钮，财务确认后执行。**最保守**，可作为 1/2 的兜底。

**建议**：先上 **方案 2（财务自判）+ 方案 3（人工兜底按钮）**，无需节拍器改动即可闭环；方案 1 作为后续优化（更实时）。

## 7. 边界与规则

| 场景 | 规则 |
|---|---|
| **分批收货** | 收货途中只更新 received_payable 显示，不红冲；收齐后一次性终态核销。 |
| **让步接收 concession** | 计入合格实收（按实收量×单价），全额或按让步价——**让步价从哪来需财务确认**（当前无让步价字段，默认按原单价×实收量，或人工调）。 |
| **拒收 reject** | 不计入 received_payable；若该单只有拒收、无合格实收 → received_payable=0 → 应付全额红冲（退货不付）。 |
| **超收** | 不自动补记，进「超收待审批」；审批放行后按实收补记 journal。 |
| **跨订单合并采购单（B3 场景）** | 一张 fin_po 服务多订单：按整单实收核销即可（应付本就是对供应商整单）；订单级份额拆分不在应付层做。 |
| **无价版采购单** | unit_price 为空 → received_payable 算不出 → 不参与自动核销，保持人工登记额，页面标「无价·手工」。 |
| **币种/汇率** | 应付以 fin_po.currency 记，核销同币种；跨币种沿用现有 GL 汇率口径，不在本设计引入新汇率逻辑。 |
| **已付款后才发现短装** | 已付 > received_payable → 不能红冲已付部分；生成「应向供应商追回」应收挂账或负债冲减，走人工 + journal 留痕（提示财务向供应商索赔/下单抵扣）。 |

## 8. UI 变更

1. **应付页 / 采购单工作台**：每张单显示 订购应付 / 实收应付 / 已登记 / 差额 四列 + 核销状态徽章（provisional / 已核销短装 / 超收待审）。
2. **付款界面**：付款额 > received_payable（非超收已批）→ 红色拦截提示「超过实收应付，请确认或走超收审批」。
3. **超收待审批**：新增一个轻列表（复用现有 pending_approvals 或独立），财务放行后补记。
4. A 止血的短装提示保留，B 上线后升级为「已按实收核销 ¥X」。

## 9. 迁移与灰度

1. 加列 migration（§5）+ cost_items.fin_po_id 存量回填脚本。
2. Feature flag `RECEIPT_RECONCILE`（off→只显示不核销 = 现状+A；on→自动红冲/闸）。
3. 灰度：先 on 显示（订购/实收/差额四列 + 付款闸提示），观察一周对账吻合；再 on 自动红冲。
4. 回滚：flag 关回 off，已生成的红冲 journal 保留（是真实账，不回滚账，只停自动化）。

---

## 10. 风险

| 风险 | 缓解 |
|---|---|
| "收齐"误判导致提前红冲 | 用短装容差 + 人工兜底按钮；方案 1 终态事件更准 |
| received_qty 含拒收（历史脏数据） | 节拍器 P1 #4 已修排除 reject；财务侧核销再按 inspection_result 二次过滤 |
| 已付款后红冲造成账不平 | §7 最后一行规则：不冲已付，转应收/索赔挂账，全走 journal |
| cost_items 未绑 fin_po | 登记入口回填 fin_po_id + 存量脚本；未绑的退化为人工 |
| 让步价缺失 | 上线前财务确认让步计价口径,或加让步价字段 |

1. **让步接收**按原单价还是让步价计应付？（缺字段则需加）

4. "收货完成"信号：先上**方案 2+3**（财务自判+人工），是否可接受滞后？还是要节拍器补**方案 1** 终态事件？
